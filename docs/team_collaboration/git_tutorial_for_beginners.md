# Aiedio 项目 Git 协作与最佳实践指南 (小白团队专供版)

这是为咱们 5 人小队量身定制的 Git 极简工作流。这套流程抛弃了企业级复杂的“切分支 -> 冲突合并 -> 提 PR”流程，专攻**“无脑、安全、防覆盖”**。

---

## 💡 核心定心丸：为什么你的“旧文件”绝不会覆盖别人的“新代码”？

很多同学最怕的是：*“我的电脑好几天没联网了，其他目录的代码都是旧的。我这时候推送代码，会不会把我本地旧的内容，强行盖满云端，把你刚写的新代码清空？”*

**答案是：绝对、绝对不会！** 
Git 的本质不是“上传打包好的压缩包覆盖云端”，而是“精确到行的记账本（打补丁）”。
1. 当你点 commit（提交）的时候，Git 只会记一笔账：“李某某今天在此图中，往 client/app.vue 第 10 行加了一个按钮，**其他啥都没碰**”。
2. 因为你压根没碰过 ackend 目录，当你推送上云端时，Git 在底层根本看都不会看 ackend 一眼。
3. **安全拦截**：最坏情况，如果你没同步最新状态就硬推，云端的门卫会直接把你拦截弹窗报错（etch first），要求你必须先执行 git pull。一旦 pull 了，电脑会自动把你队友最新的服务端代码“拼”进你的文件夹，完好保留你的前端界面，随后即可安全推上云端。

## 🛠️ 我们团队的“Git 极简三板斧”日常操作流

你们每次写代码、交作业，只需要把这三个步骤贴在屏幕边缘。**永远在一个 master 分支开发，绝不切其他分支。**

### 步骤 0：每天开工第一件事（写代码前的热身）
每天你想坐在电脑前开始写你负责的目录时，**不要急着改代码，先敲这行**用来同步别人昨晚熬夜肝的心血：
`ash
git pull origin master
`
*（如果你刚开 VS Code，也可以点左侧“源代码管理”图标，点菜单里的 拉取/Pull）*

### 步骤 1：闭关苦干，绝不乱杀
在自己的负责目录（如仅在 client 或 	ester）里去编写、调试你的代码。**注意：千万不要好奇去改别人的文件夹**，那是你们唯一的安全边界！

### 步骤 2：封装自己的补丁（第一板斧 + 第二板斧）
代码写好了，想要交上去了，在终端顺次敲：
`ash
git add .
# 注意：引号里面写几句人话，不要写 asdfghjkl，如实描述加了什么功能。
git commit -m "完成前端的一个按钮测试"
`

### 步骤 3：准备登机，融合大家的成果（第三板斧）
刚才由于你可能写了几个小时，队友可能已经抢在你前面把别的模块更新上去了。在登机（推送）前，最后拉取一次，防止门卫把你拦下：
`ash
git pull origin master
`
*说明：因为咱们物理隔离了四个目录。Git 此时就像变魔术，会自动把云端的新东西和你今天写的补丁无缝融合在一起，绝对没有任何冲突报错。*

### 步骤 4：发射上云（第四板斧）
安全融合后，最后一步：
`ash
git push origin master
`
大功告成！今天你的作业进度已经百分百存入老师和队友都能看见的 Github 主仓库里了。

## ⚠️ Aiedio 项目强制“家规”（Best Practices）

1. **绝对禁区原则**：
   本项目从根目录物理划分为：
   - 🎨 Client联队 -> 只准碰 client/
   - ⚙️ Backend中枢 -> 只准碰 ackend/
   - 🧠 Core引擎 -> 只准碰 core_engine/
   - 🕷️ Crawler爬虫 -> 只准碰 crawler/
   **跨目录的帮助请在群里“口头教”，绝对不准跨物理目录越权修改别人的文件**（否则会出现可怕的 Git Conflict 冲突需要手工修补）。

2. **多用 Git 终端，少点不知名的 IDE 报错按钮**：
   如果你遇到了自己看不懂的英文报错，**不要随便点弹窗里的 OK 或者 Yes 去强制做某事**，把红色错误英文复制发给大模型（如 Github Copilot），问问它怎么解，一般仅仅是因为你忘了敲一次 git pull 导致的。
